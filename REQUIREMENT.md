## 客户地址信息维护
1. 地址信息包括 地址ID，客户ID， 地址类型，地址详情，修改时间(日期类型)，是否是通讯地址(Y-是，N-否)，是否是最新地址(Y-是，N-否) 字段
2. 一个客户可以有多个地址信息
3. 只能有一个是通讯地址(某个客户的所有地址中只有一个地址的 是否是通讯地址 标识为 Y，其余的为N)
4. 每种类型的地址可以有多个，但只能有一个是最新地址(比如客户有多个 居住地址，这些居住地中只有一个 是否是最新地址标识为Y，其余的为N)
5. 通讯地址是最新的(如果一个地址是通讯地址，则其是否是最新地址标识一定为Y)
6. 接口每次上送的是一个地址数组(例如Java中的 List<CifAddress>，可能是增量也可能是全量的，这里都按增量处理)，
   里面每个地址可能会标注该地址是否为通讯地址，是否为最新地址，若没有标注，则其是否为通讯地址为N，是否为最新地址为Y，修改时间 为当前系统日期
   若有重复地址(以 联系地址 为例)，即上送的联系地址有多个，但若其地址类型和地址详情一样则认为是一样的，可以合并为一个，合并规则为
   修改时间为当前系统日期；是否是通讯地址 若其中一个是Y,则为Y，否则为N；是否是最新地址 若其中一个是Y,则为Y，否则为N
7. 客户的存量地址数据(如数据库中，内存中等)，里面每个地址也可能会标注该地址是否为通讯地址，是否为最新地址，若没有标注，则其是否为通讯地址为N，是否为最新地址为N
   若有重复地址(以 联系地址 为例)，即存量的联系地址有多个，但若其地址类型和地址详情一样则认为是一样的，可以合并为一个，合并规则为
   修改时间为当前系统日期；是否是通讯地址 若其中一个是Y,则为Y，否则为N；是否是最新地址 若其中一个是Y,则为Y，否则为N，合并完后，只保留合并的，其他的删除
8. 若接口上送了通讯地址为是的地址，挑选出这些通讯地址，按 其他地址、联系地址、居住地址、单位地址、户籍地址、证件地址、营业地址、注册地址、办公地址、永久地址 顺序选择通讯地址作为此客户的通讯地址
9. 若接口没送通讯地址为是的地址，则以存量地址信息中最后修改的是否为通讯地址为Y的地址为通讯地址(即在存量地址中挑选出所有是否为通讯地址为Y的地址，取修改时间最大的，修改时间为空的默认为其修改时间最小)
9. 若存量地址信息中也没有通讯地址则依次选 其他地址、联系地址、居住地址、单位地址、户籍地址、证件地址、营业地址、注册地址、办公地址、永久地址中最后修改的地址作为通讯地址(
	即若有 其他地址 则选最近修改的其他地址，若没有但有联系地址，就选最近修改的其他地址联系地址，依次类推;
	先从接口上送的地址中选，若接口没有上送，再从存量地址中选
	修改时间为空的默认为其修改时间最小
	)
10. 对于某种类型的地址，若接口中有上送，选出是否是最新地址标识为Y的是这种类型地址的最新地址(即若接口上送了联系地址，则次联系地址为最新的联系地址，若送了多个联系地址，取第一个为最新)
11. 对于某种类型的地址，若接口中没上送，则以存量地址信息中最后修改的是否为最新地址为Y的地址为最新地址(
	即若接口没有上送 联系地址，但存量数据中有 联系地址 ，挑选出 是否为最新地址标识为Y，取修改时间最大的，修改时间为空的默认为其修改时间最小，
	同类型其他地址的是否为最新地址标识为N
	)
12. 对于某种类型的地址，若存量地址中也没有最新的，则选最近修改的为最新地址(
	即接口没有上送某一类型的地址，如没有上送 联系地址，此时分两种情况
	1.存量地址中也没有此种类型的地址，若不处理
	2.存量地址中有此种类型的地址，但没有 是否为最新地址标识为Y，就取修改时间最大的，修改时间为空的默认为其修改时间最小
	)
13. 若接口有上送的地址 其地址类型和地址详情一样则认为是对地址进行更新(具体规则为，这里以 联系地址 为例
	1. 若接口上送了联系地址，若存量地址中没有 联系地址，那这个联系地址就是新增的
	2. 若接口上送了联系地址，若存量地址中有联系地址，但其地址类型和地址详情不一样，那这个联系地址也是新增的
	3. 若接口上送了联系地址，若存量地址中有联系地址，其地址类型和地址详情一样，那这个联系地址是用来更新存量联系地址的
    )

# 业务场景
客户第一次做业务，进行客户地址维护，此时上送的地址都为新增，安上述规则入库(数据库or内存)，入库后即成为存量地址
客户第二次做业务，进行客户地址维护，此时上送的地址可能是新增，也可能是更新之前的存量地址，具体以规则判断，进行维护

# Service 参考实现算法
1.使用两个数组(如java中的List)分别表示上送的地址数组和存量地址数组
2.对这两个数组分别进行合并操作，按合并规则进行。对上送地址数组，删除掉重复的，只保留合并后的
  对存量地址数组，保留所有地址，对合并后需要删除的地址，将其del_flag设置为Y,del_flag为Y不参与业务规则
3.应用更细/新增规则，对于新增的地址，上送数组中地址的seqNo为空，对于更新的地址，上送数组中地址的seqNo设置为对应存量地址的seqNo
4.对这两个数组应用 通讯地址和最新地址规则，挑选出所有通讯地址，和每种地址的最新地址
5.将这两个数组中的标识(是否通讯地址，是否最新)都设置为N
6.将挑选出的通讯地址的标识设置为Y
7.将挑选出的最新地址的标识设置为Y
8.对上送地址数组中seqNO为空的地址，生成id，然后批量insert
9.对上送地址数组中seqNO不为空的地址，根据其seqNO，更新存量地址中对应的数据，然后批量update

# Data Model
实体类 `CifAddress` 包含字段：
- `seqNo` (String 地址ID)
- `clientNo` (String 客户ID)
- `addressType` (String 地址类型)
- `addressDetail` (String 地址详情)
- `lastChangeDate` (Date 修改时间)
- `isMailingAddress` (String 是否通讯地址，'Y'/'N')
- `isNewest` (String 是否最新地址，'Y'/'N')
- `del_flag` (String 逻辑删除标识 Y-删除，N-正常)

技术栈：
	Java8 + Maven
	不要新建异常，使用 RuntimeException
	不用数据库，使用内存代替，但不要使用 Map
